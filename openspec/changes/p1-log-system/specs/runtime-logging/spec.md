## ADDED Requirements

### Requirement: 内核日志全量落盘
内核在扩展进程启动时，应当（SHALL）从最早即订阅日志并写入 App Group 容器内的日志文件，使主 App 经 WebSocket 订阅 `/logs` 之前产生的日志（尤其订阅 proxy-provider 的下载日志）也被完整记录。落盘不得（MUST NOT）依赖主 App 是否已连接 controller。

#### Scenario: 内核启动即落盘
- **当** 内核在扩展进程启动
- **则** 从启动最早开始，日志被写入 App Group 容器内的日志文件

#### Scenario: 捕获 WS 订阅前的日志
- **当** 内核在 controller 就绪前下载订阅 proxy-provider
- **则** 该下载日志出现在落盘文件中，即使主 App 此时尚未连上 controller

### Requirement: 日志文件滚动限容
日志文件不得（MUST NOT）无限增长。系统必须（MUST）按单文件大小滚动、保留有限个文件，并使日志总大小封顶（如 5 MiB）。

#### Scenario: 单文件达上限滚动
- **当** 当前日志文件大小达到单文件上限
- **则** 滚动到新文件继续写入

#### Scenario: 总量封顶删旧
- **当** 日志文件数量或总大小达到上限
- **则** 删除最旧的文件，使总大小不超过封顶值

#### Scenario: 重启不累积
- **当** 内核多次重启
- **则** 日志总大小仍受封顶约束，不随重启次数无限增长

### Requirement: 实时日志显示带时间戳且可复制
主 App 监控页应当（SHALL）实时显示日志，每条带接收时间戳（mihomo `/logs` 不含时间，采用主 App 接收时刻），且日志文本应当（SHALL）可选中复制。

#### Scenario: 显示带时间戳
- **当** 主 App 经 WebSocket 收到一条日志
- **则** 该日志以「时间戳 + 级别 + 内容」形式显示

#### Scenario: 选中复制
- **当** 用户在日志区长按选中文本
- **则** 可复制选中的日志内容

### Requirement: 暂停与继续不丢日志
监控页应当（SHALL）支持暂停日志滚动以便查看与复制；暂停期间内核产生的日志不得（MUST NOT）丢失。

#### Scenario: 暂停冻结画面
- **当** 用户点击暂停
- **则** 当前日志画面冻结，便于查看与复制

#### Scenario: 暂停期间不丢日志
- **当** 暂停期间内核仍持续产生日志
- **则** 这些日志被保留，继续后可见

#### Scenario: 继续恢复滚动
- **当** 用户点击继续
- **则** 恢复实时滚动，并补上暂停期间累积的日志

### Requirement: 高频日志性能与内存上限
高频日志场景下，UI 刷新必须（MUST）以节流批量方式进行，避免逐条刷新导致卡顿；内存中仅保留最近 N 条，完整历史以落盘文件为准。

#### Scenario: 节流批量刷新
- **当** 日志高频产生
- **则** UI 以批量节流方式刷新，而非每条触发重绘

#### Scenario: 内存条数上限
- **当** 内存中日志条数达到上限 N
- **则** 丢弃最旧的内存条目，但完整历史仍保留在落盘文件中

### Requirement: 完整日志回看
主 App 应当（SHALL）提供从落盘文件回看全量日志的能力，含内核启动最早段与历史会话，并支持按关键词搜索。

#### Scenario: 回看全量含最早
- **当** 用户打开完整日志回看
- **则** 显示落盘文件中的全量日志，含内核启动最早一段

#### Scenario: 跨会话回看
- **当** 用户回看日志
- **则** 可见滚动限容范围内的历史会话日志

#### Scenario: 搜索过滤
- **当** 用户输入搜索关键词
- **则** 仅显示匹配关键词的日志行

### Requirement: 日志导出分享
主 App 应当（SHALL）支持将完整日志文件经系统分享导出，便于诊断与存档。

#### Scenario: 导出分享
- **当** 用户点击导出
- **则** 调起系统分享，可将日志文件发送或保存
