# 代码质量复盘与 Review Checklist

> 适用范围：本项目所有代码改动；内容为通用软件工程实践，不限本项目。
> 来源：`p1-subscription-as-config` 的 code-review 复盘（2026-06）——一次 review 在自测全绿的代码里抓出 2 个严重缺陷（含一个会致真机 iOS 推送断裂、却能骗过自测的隐藏 bug）。

## 核心原则

**几乎所有 review 抓到的缺陷，根因都是「未经验证的假设被当成了事实」**——假设库的行为、假设没有现成方案、假设测试有效、假设解耦不破坏不变量。

> 防范的核心只有一件事：**把假设显式化、再验证化。**
> 元原则：**实验 > 推理 > 直觉。** 关键假设用一个最小实验去证伪，而不是「我觉得应该是这样」。

## 七类常见陷阱（现象 → 根因 → 防范）

### 1. 用第三方库 / API 前，不验证其真实行为就假设
- **案例**：以为某解析函数返回「调用方原样」（空字段=未提供），实际它以一份「默认配置」为基底**预置了默认值**，导致 `len == 0` 的守卫永不成立、整段自定义逻辑变成死代码。
- **根因**：基于想当然 / 记忆 / 文档措辞使用依赖，而非实证。
- **防范**：集成任何第三方能力前，先写一段**探针**（最小实验或读源码）确认其边界行为——默认值、空输入、错误返回、副作用。「我以为它会 X」必须降级为「我验证过它会 X」。

### 2. 造轮子前不调研「框架是否已经提供」
- **案例**：自己实现一套缓存落盘逻辑，结果底层框架早已自动提供等价能力，自造版本反而更弱、还覆盖了上游的正确配置。
- **根因**：先动手实现，没先问「这个需求框架是不是已经解决了」。
- **防范**：写任何「基础设施级」代码前先搜上游 / 标准库的现有能力。**默认假设「常见需求大概率已有解」**，证伪后再自己写。

### 3. 测试断言「实现细节」而非「业务契约」→ 假绿
- **案例**：某校验逻辑的测试只断言「列表非空」，而依赖的默认值本就非空，于是**无论代码对错测试都绿**，完美掩盖了缺陷。
- **根因**：测试测的是「代码碰巧做了什么」，不是「业务真正要什么」。
- **防范**：测试断言**可观察的业务行为**（如「列表里必须含某个关键项」），不是实现量（长度 / 调用次数）。写完每个测试问一句：**「如果实现是错的，这条测试会变红吗？」** 答不上来就是假绿。可先把实现改坏一下，确认测试确实会红（红-绿）。

### 4. 用字符串启发式（子串 / 正则）做语义判断
- **案例**：用 `contains('proxies')` 判断内容是否为合法配置 → 一个含该字样的 HTML 错误页被误判通过。
- **根因**：拿词法工具解决语法 / 语义层的问题。
- **防范**：语义判断交给**正规解析器**（YAML/JSON parser、AST）；`contains` / `regex` 只配做词法层的事。脆弱的启发式迟早被边界数据击穿。

### 5. 重构改变数据流时，不排查「断裂的隐含不变量」
- **案例**：把「获取」与「使用」两步解耦后，原本隐含的「使用的就是当前输入对应的数据」这一不变量断裂 → 输入变了却用了旧数据，且无任何提示。
- **根因**：拆分时只看「功能还在不在」，没问「原来什么是耦合默默保证的」。
- **防范**：解耦 / 拆分的 review 专门问：**「这两件事原来绑在一起时，谁在保证某个一致性？拆开后还有谁保证？」** 断裂的不变量要显式补回。

### 6. 作者看不见自己的盲点 —— 独立、对抗性 review 不可省
- **案例**：上述缺陷作者自测全绿、自读代码也未发现，是**多个互相独立、各执一个视角（正确性 / 边界 / 设计）的审查者**抓出来的。
- **根因**：「知识诅咒」——人验证不了自己脑中的错误前提，因为他以为它对。
- **防范**：关键代码必须有**独立视角**的 review；凡「我以为 / 应该 / 想当然」的假设，要被外部主动挑战。**多个互相独立的检查 > 一个全能检查。**

### 7. 断言「某能力在某平台无效 / 不支持」却无视成熟同类实现已有它
- **案例**：设计运行参数时，断言 `allow-lan`「在 iOS NE 是死开关」并据此排除——但成熟同类 **clashmi（同样跑 iOS NE）恰恰暴露了 allow-lan + mixed-port**。用户一句「clashmi 为什么有这个开关」即戳破：它在 iOS 上能用（NE 进程可开监听 socket、接受局域网入站＝iPhone 当局域网代理网关，Shadowrocket/Stash/clashmi 皆如此）。我把「我以为不行」当成了平台事实，还无视了眼前的反证。
- **根因**：对平台能力下「不支持/无效」结论时未验证；更糟的是**无视了「成熟同类产品已经实现」这一现成反证**。
- **防范**：下「某能力在 X 平台不可做/无效」结论前，先问 **「有没有成熟的同类实现已经做了这件事？」**——若有，它就是反证，必须先查清**它怎么做到的 / 机制是什么**再决定，绝不能用「我以为不行」去排除一个功能。能力的「有/无」只接受两种证据：真机/源码验证，或同类实现佐证。

### 附：成本意识与范围克制
- 加复杂度前掂量成本 / 收益；「技术最优」要算落地成本（体积、跨层、维护）。本项目曾差点为「内核级最权威校验」把整个内核库（数十 MB）链入主程序，及时改用成本极低、严谨度等价的方案。
- 下重手（架构级改动、引依赖、大重构）前先确认范围，能在更轻的层面等价解决就不下重手。

## Review / 编码 Checklist（每个时机各问一句）

| 时机 | 必问 |
|------|------|
| 用库 / API | 这个行为我**验证过**还是**猜的**？（默认值 / 空值 / 错误返回 / 副作用）|
| 写新基础设施 | 框架 / 上游是不是**已经提供**了？ |
| 写测试 | **如果实现是错的，这条测试会红吗**？它测的是契约还是实现？ |
| 做校验 / 解析 | 我在用**解析器**，还是在用 `contains` / regex 凑？ |
| 重构 / 解耦 | 原来耦合**默默保证**的不变量，拆开后**谁来保证**？ |
| 加复杂度 | 成本（体积 / 跨层 / 维护）算过了吗？有没有**更轻的等价解**？ |
| 断言平台不支持 | 有没有**成熟同类**已经做了这件事？若有，先查它**怎么做到的**，别用「以为不行」排除功能 |
| 宣称完成 | 有**独立视角**挑过吗？我哪些结论是「以为」而非「验证」？ |

## 如何使用

- **写代码时**：对照上表自查，尤其「用库 / 写测试」两栏。
- **提交前**：跑通质量门禁（见 `CLAUDE.md` §质量门禁）+ 自问「宣称完成」一栏。
- **review 时**：把六类陷阱当作审查维度逐一过；对作者的「假设」主动证伪。
- **复盘时**：新发现的同类教训追加到本文件，保持其为「活的」经验库。
